# Q: 「教育部版 Jamf Pro」與「校方自購版」有什麼不同？可以互通嗎？

# Answer

兩者在功能權限、更新頻率與「新功能開啟控制權」上存在顯著差異，但資產可以在兩者間轉移。

## 帳號權限三層級

在「教育部版 Jamf Pro」的系統架構中，權限依據行政管理與系統安全需求，分為以下三種角色位階：

1.  **教育部管理人員 (Full Admin)**：擁有 Jamf Pro 實例 (Instance) 的最高管理權限。負責制定一致的管理政策、開啟功能開關（如 DDM 進階 Payload）。
2.  **經銷商/廠商帳號**：具備輔助多校進行大規模部署的權限，許多權限位階高於校方，但仍無法越權更改教育部設定的核心限制。
3.  **校方管理帳號**：擁有最貼近教學現場的日常維運權限（如 App 派送、智慧群組、裝置重新命名等）。但在功能細部上有安全性限制，例如：**無法刪除描述檔**。這是一項基於安全政策的防護措施，旨在確保「教育部強制部署」的關鍵設定不被意外移除。

## 核心差異比較

| 項目 | **教育部版 (校方帳號)** | **校方自購版 (Full Admin)** |
| :--- | :--- | :--- |
| **最高管理權限** | **教育部核心團隊** (政策管理者) | **校方 100% 自主掌控** |
| **伺服器架構** | **Jamf 原廠代管 (雲端)** | **校務專屬實例 (雲端)** |
| **描述檔管理** | **受保護**。無法刪除，確保核心政策不毀損 | **無限制**。可自由刪除與測試描述檔 |
| **軟體更新 (DDM)** | **目前未支援**。權限未連動且無排程 | **全功能**。可自訂強制的更新死線 (Deadline) |
| **自動化與 API** | **限制較多**。無法自行上傳腳本或介接 API | **完全開放**。支援所有自動化與自訂腳本 |
| **新功能測試** | 須由教育部人員手動登入總後台開啟 | 釋出當天即可隨時自行手動開啟 |

## 為什麼功能看得到卻不能用？（行政與技術的真相）

資訊組長在管理後台常會發現某些新功能圖示（如 iOS 26 的一些新管理 Payload）雖然出現，但點選後無法生效，其主因如下：

1.  **權限層級與總開關**：許多新功能（如 Platform SSO 配置、DDM 診斷頻道）其「核心總開關」位於系統的全域設定頁面。校務帳號雖然是現場的維運主力，但這些「總開關」僅限具備最高層級的教育部管理人員才有權限勾選開啟。
2.  **行政維穩優先**：中央管理人員的主要職則在於維護全台海量設備的資安底線。因此，在沒有全台一致的教學需求或政策變更前，管理人員通常不會特地登入後台針對新功能進行改動或測試，這導致新機型即使具備新功能，也會因為管理端的「全域設定未開」而無法使用。
3.  **防止政策斷層**：校方之所以無法刪除描述檔，是為了避免校方在排錯過程中，不慎將教育部指定的「資安過濾」或「設備監管」設定檔移除。若遇描述檔衝突，通常建議透過修正規範對象，也就是調整派送範圍 (Scope) 取代直接刪除。
